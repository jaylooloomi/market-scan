# Market Scan 優化建議 —— 產品經理視角(使用者 × 使用流程)

**作者視角**:資深產品經理(消費級 + 專業級金融工具)
**日期**:2026-06-01 ｜ **版本**:v2(經 `pm-reviewer` agent 審查後修訂,評分 B− → 見文末修訂紀錄)
**方法**:Jobs-to-be-Done + 使用者旅程 + 摩擦點盤點

> 一句話:**價值主張很強(比市場早抓題材),但目前的使用流程把大多數人擋在門外,留下來的人也沒有「該不該信」的依據。** 優化重點是 onboarding 摩擦 與 信任建立,不是加功能。

---

## 1. 目標使用者(personas)

| Persona | 是誰 | 核心需求(JTBD) | 技術力 |
|---|---|---|---|
| **A. 進階散戶/兼職投資人** | 自己選股、會看新聞、想比別人早 | 「幫我每天過濾雜訊,告訴我哪個**還沒漲**的題材值得研究」 | 中(裝得起,但嫌煩) |
| **B. 重度散戶研究員 / 個人投顧** | 自己做研究、或服務少數客戶 | 「給我有**結構**(因果樹+歷史對應)的研究起點,省蒐集時間」 | 中高 |
| **C. 財經內容創作者/KOL** | 寫電子報、拍片 | 「給我**有梗、有歷史對照**的題材素材」 | 中 |
| **D. 量化/開發者** | 寫策略、接訊號 | 「把**結構化 verdict**(JSON)餵進我的回測/選股 pipeline」 | 高 |

**洞察**:A 是最大群、技術摩擦最高;B/D 最可能付費;C 是天然擴散管道。**v1 應為 A 降摩擦、為 B 建信任。**
- **D 的對應旅程(原本漏掉)**:D 不走報告 UX,走的是「`run_daily()` 回傳的 verdicts JSON / `--db` 的 SQLite」→ 痛點是**沒有穩定的程式介面契約**(schema 會不會變?有沒有 versioning?)。對 D 的產品動作是「凍結 verdict schema + 出一份 API/schema 文件」,不是報告美化。
- ⚠️ 不誇大:這是 CLI plugin,需要 Claude Code 訂閱 + terminal;B 不該寫成「家族辦公室」(那層級多半有專業終端與團隊,非本產品 ICP)。

## 2. 核心使用情境(scenarios)

| 情境 | 觸發語(已存在於 `dig.md`/`SKILL.md`) | 使用者真正想要 |
|---|---|---|
| **S1 每日早報** | 「今天有什麼?」/ 排程 06:00 | 起床 2 分鐘掃完:有沒有「還沒發酵」的新題材 |
| **S2 主題研究** | 「幫我研究矽光子」 | 因果樹 + 歷史對應 + 現在數據 |
| **S3 突發事件追** | 「最近 XX 事件,台股誰受惠?」 | 受益族群地圖 |
| **S4 回溯驗證(信任情境)** | (目前無一級入口) | 「它過去抓得到嗎?」自己驗證、建立信任 |

→ **S4 是「信任情境」**。關鍵更正:**回測能力其實已存在**(`run_recall_suite`/`run_backtest_case`,[backtest.py](src/market_scan_mcp/reviewer/backtest.py);以及 `market-scan-validator` CLI)。所以 S4 不是「從頭開發」,而是「**把既有回測模組包成 `/dig replay <題材>` 指令**」——成本約 1-2 天,**因此 S4 應升到 P0**(與 demo 並列)。

## 3. 使用者旅程 + 摩擦點(Persona A)

```
發現 → 安裝 → 設定 → 首次使用 → 看報告 → 判斷該不該信 → 回訪
  😐     😣      😣        🙂          😐           😨            ❓
```

| 階段 | 現況 | 摩擦 |
|---|---|---|
| 安裝 | `pip install -e .` + 裝 plugin | 要會 terminal/git;非開發者卡關 |
| 設定 | 申請 FinMind token、建 `.env` | **最大棄用點**:第三方註冊 + 找 token + 編檔 |
| 首次使用 | 自然語言 / `/dig today` | 不知道「該問什麼」 |
| 看報告 | 4 區中文 markdown | **資訊架構過載**:因果樹三階十幾檔 ticker,新手不知道「我該查哪一檔」 |
| **判斷該不該信** | **無任何依據** | 😨 **致命**:沒有歷史命中率;`anomaly_score` 是規則式分數、不是後驗命中率 |
| 回訪 | 無提醒、無個人化、同題材重複出現也無「已追蹤 N 天」標記 | 容易忘、且會懷疑系統卡在同一訊號 |

## 4. 產品優化建議(依使用者價值排序)

| 優先 | 建議 | 現況 vs 要做 | 解哪個痛點 |
|---|---|---|---|
| **P0** | **降 onboarding 摩擦** | `--demo` flag **已存在但預設關**([daily_cli.py](src/market_scan_mcp/daily_cli.py))。**要做的是**:首跑偵測到無 token → 自動提示/切 demo;加 `market-scan setup` token 精靈 | 安裝/設定棄用 |
| **P0** | **S4 回溯變一級功能** | `/dig replay <題材>`——**包裝既有 `backtest.py`,非從頭做**(1-2 天) | 信任 |
| **P0** | **信任層** | 報告附該訊號的歷史佐證。**注意區分**:`lag_days`(已存在於 themes.json,**現在就能顯示**)vs `hit_rate`(需 live 累積,見 §5) | 「該不該信」 |
| P1 | **報告漸進式揭露** | 預設只看 🟢 強訊號 3 行摘要 + 「先看哪一檔」建議,點開才看全因果樹 | 資訊過載 |
| P1 | **關注清單**(v1.5,原 v1 不做) | 「我關注 X,有相關題材就標記」 | 回訪勾子 |
| P2 | **推播** | `reporting/telegram.py` **已存在**;且 Telegram **不在** HANDOFF「已確認不做」清單(那只列 Email/LINE)→ 沒有約束衝突。但要計入**自身 onboarding 成本**(用戶要開 Bot、取 chat_id,不比 FinMind 容易) | 回訪 |

## 5. 產品成功指標(使用者視角)

| 指標 | 定義 | **可量測時間點** |
|---|---|---|
| **Time-to-first-insight** | 安裝 → 第一份有用報告 | ✅ 現在(可用 demo 模式錄製) |
| **(過渡)Phase 0 佐證分布** | 15 個歷史測試點的 STRONG/TOO_LATE 分布 | ✅ 現在(已有數據) |
| **信任轉換率** | 跑過 S4 → 持續使用比例 | ⏳ S4 上線後 |
| **每日掃描留存(D7/D30)** | 早報情境回訪率 | ⏳ 需用戶量 |
| **hit_rate / 平均領先天數** | 推薦後 N 天的真實命中 | 🔴 **live 上線 T+90 天後**才有第一筆(`metrics.py` 明載「cannot be fast-forwarded」) |

> ⚠️ **關鍵更正**:`hit_rate`、信任轉換率都不是近期可追蹤的 KPI。要驅動近期決策,只能用 ✅ 標的兩個現成指標 + Phase 0 既有分布做過渡,別把 T+90 的指標當成上線就能看的儀表板。

## 6. 漏掉的使用者觀點(本版補上)

- **FinMind 額度耗盡**:免費版 600 req/hr,密集使用(全市場抓單日)會撞限 → 中度使用者的隱形痛點。
- **同題材重複出現**:連幾天看到同一強訊號卻無「已追蹤 N 天」標記 → 信任損傷。
- **Persona C 的引用/著作權焦慮**:KOL 轉述報告時,需知道來源(RSS 轉述 / 爬蟲 / FinMind 授權)可否公開引用。
- **退出旅程**:停用/刪 plugin 時有沒有「最後一次正向互動」——留存洞察來源。

---

## 7. 結論

> **Market Scan 不缺功能,缺的是「讓人敢用、敢信」。** 三件最重要:① 把 `--demo`/設定精靈做好(注意:這是**要做**的功能,不是現成預設);② 把**回溯驗證**包成一級指令(`backtest.py` 已有,成本低,應 P0);③ 在報告裡放**現在就能顯示的 lag_days**,並把 `hit_rate` 誠實標為 T+90 後才有。信任不能等——沒有它,再準的訊號使用者也不會照做。
